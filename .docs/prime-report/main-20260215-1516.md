## Project Overview

### Purpose
**BeKesher** (בקשר - "In Touch") is a social networking platform designed specifically for new immigrants (olim) in Israel. It facilitates meaningful connections through weekly automated group matching, gamified social interactions, and structured meetups.

### Core Value Proposition
- Help new immigrants overcome social isolation through structured, gamified networking
- Match participants based on region, age, interests, and values (5-stage progressive matching algorithm)
- Weekly task-based meetups with 2-4 participants per group
- Subscription-based access (transitioning to monthly seasons at 100 NIS/month)
- Support for Russian (primary) and English languages with UI language switching

### Application Type
Full-stack SaaS platform with:
- **User-facing**: Telegram Mini App (React + Vite)
- **Admin-facing**: Web Dashboard (React + Vite)
- **Backend**: Convex serverless platform (real-time database + functions)

### Current State
- **Version**: MVP v1.0 (post-implementation phase)
- **Development Stage**: Production-ready with active deployment
- **Recent Focus**: i18n implementation, UI styling improvements, authentication layer enhancements
- **Next Milestone**: Beta testing launch (Feb 18), official season launch (Mar 15)

### Architectural Patterns
1. **Event-Driven Real-time**: Convex subscriptions provide live updates to all clients
2. **Domain-Driven Organization**: Code organized by business domain (participants, groups, matching, payments, etc.)
3. **Serverless Functions**: All backend logic runs as Convex queries/mutations/actions
4. **Monorepo with Workspaces**: npm workspaces manage three sub-projects
5. **Staged Algorithm Execution**: 5-stage progressive matching with relaxing constraints
6. **Webhook-Based Integration**: External services communicate via HTTP endpoints

## Sub-System Deep Dives

### 1. Convex Backend (`convex/`)

**Architecture Pattern**: Domain-driven modular functions
**Language**: TypeScript 5.x with strict Convex validators

#### Logic Hubs

| Module | Purpose | Key Functions | Pattern |
|--------|---------|---------------|---------|
| **participants.ts** | User lifecycle | `getByTelegramId`, `register`, `updateProfile`, `setStatus` | CRUD + Status FSM |
| **matching.ts** ⭐ | Weekly group creation | `runWeeklyMatching` (5-stage algorithm) | Staged waterfall |
| **groups.ts** | Group management | `create`, `list`, `updateStatus`, `getParticipants` | Lifecycle manager |
| **payments.ts** | Subscription handling | `createRequest`, `handleCallback`, `extendSubscription` | External integration |
| **feedback.ts** | Post-meetup surveys | `submit`, `awardPoints` (10 pts) | Validation + rewards |
| **support.ts** | Ticket system | `create`, `answer`, `close` | State machine |
| **notifications.ts** | Telegram messaging | `sendGroupNotification`, `sendReminder` | External action |
| **crons.ts** | Scheduled jobs | `weeklyMatching`, `closeWeek`, `paymentReminders`, `cleanup` | Scheduled functions |
| **http.ts** | Webhooks | `POST /payplus-callback`, `POST /telegram-webhook`, `GET /health` | HTTP endpoints |

#### Authentication Strategy

**Dual Authentication System** (no shared auth between user/admin):

| Context | Method | Implementation | Token Storage |
|---------|--------|----------------|---------------|
| **User (Telegram)** | HMAC-SHA256 validation | `authUser.ts`: `userQuery`, `userMutation` wrappers validate Telegram `initData` per request | Ephemeral (validated per request) |
| **Admin (Google)** | OAuth 2.0 JWT | `auth.config.ts` + `authAdmin.ts`: Convex validates Google JWT via OIDC, `adminQuery`/`adminMutation` wrappers check `ctx.auth.getUserIdentity()` | Convex `sessions` table |

#### Database Schema (6 Tables)

```typescript
participants {
  // Identity
  telegramId: string (indexed)
  name, phone, photo

  // Demographics
  birthDate: string (YYYY-MM-DD)
  gender, region ("North"|"Center"|"South"), city

  // Preferences
  targetGender, targetAgeFrom, targetAgeTo

  // Profile
  aboutMe, profession, purpose, values[], interests[]

  // Status
  status: "Lead" | "Active" | "Inactive"
  onPause: boolean
  totalPoints: number

  // Subscription
  paidUntil: timestamp, inChannel: boolean
}

groups {
  participant1, participant2, participant3?, participant4?
  status: "Active" | "Completed" | "Cancelled"
  region, createdAt (indexed)
}

feedback {
  groupId, participantId (indexed)
  rating (1-10), textFeedback, wouldMeetAgain
  photos: storageId[], submittedAt
}

paymentLogs {
  participantId (indexed)
  amount, currency, status
  payPlusTransactionId, createdAt
}

supportTickets {
  participantId?, telegramId, question, answer?
  status: "Open" | "Answered" | "Closed"
}

sessions {
  telegramId, token (indexed), expiresAt
  source: "ai" | "cicd" | "dev"
}
```

#### 5-Stage Matching Algorithm (Core Business Logic) ⭐

**Purpose**: Match active participants into groups of 2-4 weekly
**Constraints**: Region proximity, age compatibility, prevent repeats (4-week history)

| Stage | Criteria | Purpose |
|-------|----------|---------|
| **A: Strict** | Same region, ±10y age, no repeats (4 weeks) | Optimal matches |
| **B: Expanded Age** | Same region, ±15y age, no repeats | Widen age tolerance |
| **C: Allow Repeats** | Same region, ±15y age, can repeat | Accept familiar faces |
| **D: Neighboring** | Adjacent regions (N↔C, C↔S), ±15y | Geographic expansion |
| **E: Force Majeure** | Any compatible (never N+S) | Ensure no one left out |

**Flow**: Sequential waterfall → unpaired participants move to next stage → create group records → trigger Telegram notifications

**Test Coverage**: `matching.test.ts` validates algorithm parity with legacy Google Script

#### Scheduled Jobs (Crons)

| Job | Schedule | Purpose |
|-----|----------|---------|
| `weeklyMatching` | Sun 16:00 UTC (18:00 Israel) | Run 5-stage matching |
| `closeWeek` | Sat 21:00 UTC (23:00 Israel) | Mark groups completed |
| `paymentReminders` | Daily 08:00 UTC (10:00 Israel) | 3-day, 1-day, same-day warnings |
| `cleanup` | Mon 01:00 UTC (03:00 Israel) | Archive old data |

#### Authentication Flow
1. User opens Telegram Mini App → Telegram WebApp SDK injects `window.Telegram.WebApp.initData`
2. `useTelegramAuth()` extracts `telegramUser.id`
3. Every Convex query/mutation includes `initData` in headers
4. Backend `userQuery`/`userMutation` wrappers validate HMAC-SHA256 signature
5. `ctx.telegramId` populated for queries

#### State Management
**No Redux/Zustand** — all state via Convex real-time subscriptions:
- `useQuery(api.participants.getByTelegramId)` → auto-updates on backend changes
- `useMutation(api.participants.updateProfile)` → optimistic updates

```
│   └── LanguageSwitcher.tsx   # ru/en toggle
│
└── hooks/
    ├── useAdminAuth.ts        # Google OAuth + Convex auth
    └── useLanguage.ts         # i18n locale management
```

#### Authentication Flow
1. Admin visits `/login` → Google Sign-In button
2. `@react-oauth/google` obtains JWT credential
3. `useAdminAuth().login(credential)` calls `setAuth(credential)` from `convex/react`
4. Convex validates JWT against `auth.config.ts` Google OIDC provider
5. `ctx.auth.getUserIdentity()` populated, `adminQuery`/`adminMutation` check admin access
6. Session stored in `sessions` table

#### Admin Permissions
Currently **single-tier admin** (all admins have full access). Future: role-based permissions.

### 4. Internationalization System (i18n)

**Framework**: `@lingui/react` v5.9.1 (macro-based, compile-time optimized)
**Languages**: Russian (primary), English (secondary)
**Implementation Date**: Feb 15, 2026 (recent)

#### Architecture

| Layer | Implementation |
|-------|----------------|
| **Frontend** | `@lingui/react` with `<Trans>` macros, `.po` translation files |
| **Backend** | `convex/i18n.ts` exports `t()` function for notifications |
| **Compilation** | `lingui extract` → `.po` files → `lingui compile` → `.mjs` runtime catalogs |
| **Language Switch** | `<LanguageSwitcher>` component in both apps, stores preference in localStorage |

#### Workflow
1. Developer writes: `<Trans>Welcome to BeKesher</Trans>`
2. `npm run lingui:extract` → extracts to `locales/en/messages.po`, `locales/ru/messages.po`
3. Translator fills `.po` files
4. `npm run lingui:compile` → generates `.mjs` runtime catalogs
5. App loads catalog based on user preference

## Core Principles

### Code Organization

1. **Domain-Driven Modules**: Backend organized by business domain (participants, groups, matching), not by technical layer
2. **Colocated Tests**: Test files next to source files (`participants.test.ts` alongside `participants.ts`)
3. **Single Source of Truth**: Convex schema defines database structure, validators enforce types at runtime
4. **Real-time First**: All UI updates via Convex subscriptions, no manual state management
5. **Monorepo Workspaces**: Shared dependencies at root, isolated app configs in `apps/*/package.json`

### Coding Standards

**TypeScript Strictness**: Full type safety with Convex validators (`v` module)

```typescript
// Convex function signature
export const updateProfile = userMutation({
  args: {
    name: v.string(),
    region: v.union(v.literal("North"), v.literal("Center"), v.literal("South")),
    interests: v.optional(v.array(v.string())),
  },
  returns: v.null(),
  handler: async (ctx, args) => { /* ... */ }
});
```

**Naming Conventions**:
- Database indexes: `by_field` or `by_field1_and_field2`
- Internal functions: `internalQuery`, `internalMutation` prefix
- Test factories: `make*` (e.g., `makeParticipant()`, `makeGroup()`)

### Authentication Security

- **User Auth**: Telegram `initData` HMAC-SHA256 validation per request (no JWT, no sessions)
- **Admin Auth**: Google OAuth JWT validated via Convex OIDC integration
- **No Shared Auth**: Completely separate authentication flows for user vs admin

### Testing Approach

**Test Philosophy**: Focus on business logic correctness, not implementation details

**Coverage Priorities**:
1. ⭐ **Matching algorithm** (highest risk, complex business rules)
2. **Payment webhooks** (revenue-critical)
3. **Status workflows** (Lead → Active → Pause → Inactive)
4. **Business rule validation** (duplicate prevention, point awards, age calculations)

**Test Utilities**: `test.utils.ts` provides factory functions with sensible defaults:
```typescript
const participant = makeParticipant({
  name: "Test User",
  region: "Center",
  status: "Active",
  // ... all other fields have defaults
});
```

### i18n Standards

- **Russian as Primary**: All user-facing text defaults to Russian
- **Macro-Based**: Use `<Trans>` macros, not manual translation calls
- **Backend i18n**: Notifications support both languages via `t(key, lang)` helper
- **No Hardcoded Strings**: All text must come from translation catalogs

**Living Documentation**:
- `CLAUDE.md` — Project instructions for AI assistants (always up to date)
- `PRD.md` — Product requirements with architecture diagrams
- `styleguide.md` — Brand guidelines with color palette, typography, component patterns
